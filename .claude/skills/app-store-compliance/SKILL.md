# App Store compliance — BrickScan

How to keep BrickScan compliant with the Apple App Store, in code and in App Store Connect. Read the
hard rules first — they are the constraints that shape every network, permission, data-storage, or
UI change. The detailed rule mapping, the pre-submission checklist, and the running rejection log are
in `references/`.

This skill exists because BrickScan sits on top of third-party LEGO data (Rebrickable, BrickLink,
Brickset) and the LEGO trademark, which is exactly the surface Apple scrutinises. The first full audit
(July 2026) is in `docs/app-store/compliance-review.md`; remediation is tracked in issues #104 (scraping)
and #105 (phases 2-6).

## ⛔ Hard rules (never violate in an App Store build)

1. **No scraping, no hidden web views, no User-Agent spoofing.** Never load a third-party site in a
   hidden/off-screen `WKWebView` to extract data, and never spoof a browser User-Agent to defeat bot
   protection. This violates **5.2.2** (third-party content without permission — "authorization must be
   provided upon request") and **2.3.1(a)** (hidden features), and it is a standing legal/ToS risk
   regardless of review. Any new price/data source must be either (a) an official API whose ToS permit
   third-party app use — record the permission link in `references/app-review-rules.md` — or (b) a
   **visible** link-out (`SFSafariViewController`) plus manual user entry. Amazon and lego.com have no
   compliant automated price API for this use: they are link-out + manual only.

2. **No LEGO intellectual property, anywhere user- or Apple-facing.** Never put the word "LEGO" (or LEGO
   trade dress) in the app name, subtitle, keywords, app icon, splash, screenshots, or OS-facing
   permission strings (`NSCameraUsageDescription`, Siri phrases, App Intent titles). This extends to
   **any licensed IP** shown as if it were the app's own — no LEGO set renders, box art, minifigures, or
   licensed themes (Star Wars, Marvel, etc.) in the icon/splash/screenshots. The App Store description
   may carry **one** nominative mention plus the disclaimer *"LEGO® est une marque du groupe LEGO, qui ne
   sponsorise ni n'approuve cette application."* For screenshots, defer to the **`app-screenshots`** skill
   (its hard rule already forbids Rebrickable renders / box art / minifig art / LEGO logos). Guidelines:
   **5.2.1**, 4.1(c).

3. **Every new required-reason API updates `PrivacyInfo.xcprivacy` in the same PR.** Adding use of a
   required-reason API (UserDefaults, file-timestamp, disk-space, system-boot-time, active-keyboard) means
   adding/updating the matching `NSPrivacyAccessedAPITypes` entry in
   `BrickScan/Resources/PrivacyInfo.xcprivacy`. Current declared state: **UserDefaults → `CA92.1`**. Missing
   or stale manifest = upload blocked.

4. **Every new off-device data flow updates all four disclosure surfaces in the same PR.** If a change
   sends any new data off the device or stores a new data category, update together: (a) App Store Connect
   **privacy nutrition labels**, (b) `PrivacyDetailView`, (c) `PrivacyNoticeView`, (d) `PRIVACY.md`. These
   must never contradict each other or the actual network behaviour (5.1.1, 2.3). The in-app copy must name
   **every** third party contacted (today: Rebrickable API+CDN, Brickset, BrickLink, lego.com, amazon.fr,
   Apple CLGeocoder — as applicable to the shipped feature set).

5. **Permissions and Info.plist keys go through `project.yml` only.** The `Info.plist` is generated by
   XcodeGen — never hand-edit it (see `ios-build-test`). New permission strings, `ITSAppUsesNonExemptEncryption`,
   etc. are declared in `project.yml`, trademark-free.

6. **Keep `ITSAppUsesNonExemptEncryption` accurate.** It is `false` while the app uses only OS-provided
   HTTPS/Keychain. If you ever add custom crypto, redo the export-compliance determination.

7. **Run the pre-submission checklist before every submission.** Go through
   `references/submission-checklist.md` top to bottom. Do not submit on memory.

8. **After every rejection: log it, then harden.** Append the rejection to `references/rejection-log.md`.
   If it is a **new class** of failure (not already covered above), add a new hard rule to this list so it
   cannot recur.

## When adding a feature — quick gate

Before writing code that touches network, permissions, storage, or third-party content, ask:

- Does it contact a new host or third party? → hard rules 1 and 4.
- Does it use a new required-reason API? → hard rule 3.
- Does it surface LEGO/licensed IP? → hard rule 2 (+ `app-screenshots`).
- Does it need a new permission? → hard rule 5.

If any answer is yes, the compliance surfaces above must change **in the same PR**, not later.

## Reference material

## Related skills

- **`app-screenshots`** — the authority on what may appear in published visuals (no LEGO IP). Icon/splash/
  screenshot changes must satisfy it.
- **`ios-build-test`** — build via `xcodegen generate` + `xcodebuild`; the `Info.plist`/`.xcodeproj` are
  generated, never hand-edited.
- **`simulator-ui-testing`** — drive review-sensitive flows (Siri, share sheet, link-outs) in the simulator.
- **`check-rebrickable-endpoint`** — model for verifying a third-party API's real request/response shape and
  ToS before trusting code against it (apply the same rigour to any new API, e.g. BrickLink).
